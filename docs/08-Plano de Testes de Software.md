# Plano de Testes de Software

<span style="color:red">Pré-requisitos: <a href="2-Especificação do Projeto.md"> Especificação do Projeto</a></span>, <a href="3-Projeto de Interface.md"> Projeto de Interface</a>

Apresente os cenários de testes utilizados na realização dos testes da sua aplicação. Escolha cenários de testes que demonstrem os requisitos sendo satisfeitos.

Não deixe de enumerar os casos de teste de forma sequencial e de garantir que o(s) requisito(s) associado(s) a cada um deles está(ão) correto(s) - de acordo com o que foi definido na seção "2 - Especificação do Projeto". 

Por exemplo:

## Plano de Testes dos Requisitos Funcionais
| **Caso de Teste** 	| Requisito Associado 	| Objetivo do Teste 	| Passos	| Critério de Êxito | 
|---|---|---|---|---|
|**RF CT-001**  | RF-001: O sistema deve permitir que o usuário se cadastre. | Verificar a função de cadastrar usuários. |- Acessar a página de cadastro de usuário <br> - Adicionar nome de usuário, e-mail e confirmar senha       <br>  - Autenticar usuário cadastrado com e-mail e senha. | -  Será exibido na tela "O usuário foi cadastrado com sucesso". |
|**RF CT-002**  |	 RF-002: O sistema deve permitir que o usuário cadastre novas receitas com o nome dessas receitas e seus respectivos ingredientes e modos de preparo.	| Verificar a função de cadastrar uma receita, ingredientes e seu modo de preparo. | - Ser um usuário cadastrado <br>- Acessar a tela de criação de receita <br>- Adicionar as informações: Nome da receita, Ingredientes, modo de preparo e tempo médio de preparo <br>- Salvar a receita <br>- Fazer uma busca pela receita criada. <br> |  A receita foi cadastrada com sucesso.|
|**RF CT-003**  | RF-003 - O sistema deve permitir que usuários adicionem avaliações às receitas por meio de notas que variam de um número à outro (ex: 0 à 5). | Verificar se o usuário conseguirá atribuir uma nota a receita. | - Acessar a página de uma receita específica <br> - Adicionar uma avaliação com uma nota válida (entre 0 e 5) <br> - Verificar se a avaliação é exibida na receita <br> - Clicar em "Confirmar a nota" | - A avaliação será exibida na tela. |
|**RF CT-004**  | RF-004 -  O sistema deve permitir que o usuário pesquise por receitas cadastradas através de seus ingredientes ou nomes. 	| Verificar a função de busca por receitas ou ingredientes. | - Acessar a página de pesquisa de receitas <br> -  Inserir um nome de receita ou ingrediente na barra de pesquisa <br> - Verificar se as receitas relevantes são exibidas nos resultados da pesquisa <br>   | - o nome da receita ou ingredientes digitados retornam as receitas salvas no banco de dados. |
|**RF CT-005**  | RF-005: O sistema deve permitir que o usuário edite receitas já cadastradas. 	| Verificar a função de editar uma receita, ingredientes e seu modo de preparo. | - Ser um usuário cadastrado. <br> -  Acessar a tela de visulizar receita cadastradas. <br> - Alterar as informações: Nome da receita, Ingredientes, modo de preparo e tempo médio de preparo. <br> - Salvar a receita. <br> - Aceesar tela de receitas cadastradas. <br>  | -  A receita foi editada com sucesso. |
|**RF CT-006**  | RF-006: O sistema deve permitir que o usuário edite seus próprios dados | Verificar a função de editar dados do usuário. | - Ser um usuário cadastrado. <br> -  Acessar a tela de editar perfil. <br> - Alterar as informações: nome, e-mail, senha e especificação culinaria. <br> - Salvar usuário. <br> - Aceesar tela perfil do usuário. <br>  | -   Usuário foi editado com sucesso. |
|**RF CT-007**  | RF-007: O sistema deve permitir que o usuário desfaça sua avaliação 	| Verificar a função de desfazer avaliação de uma receita. | - Acessar página de uma receita específica. <br> -  Verificar a existência da avaliação feita. <br> <br>Clicar no botão de desfazer avaliação. <br> - Verificar se a avaliação feita não existe mais. <br>  - Verificar a aparição de mensagem avisando que a avaliação foi desfeita com sucesso. <br> | - A avaliação feita  não é mais apresentada, mesmo após revisitar a página da receita. |
|**RF CT-008**  | RF-008: O sistema deve permitir que o usuário exclua seu perfil 	| Verificar a funcionalidade de exclusão de perfil. | - Acessar página de perfil. <br> -  Clicar em "Excluir conta". <br> - Verificar existência de mensagem de aviso sobre o sucesso na exclusão. <br> - Verificar se houve o redirecionamento à tela de login. <br> -Verificar se, preenchendo o formulário com os dados da conta excluída, as credenciais não são reconhecidas. <br>| - Credenciais de conta excluída não são mais reconhecidas pelo sistema. |
|**RF CT-009**  | RF-009: O sistema deve fornecer ao usuário uma lista com todas as receitas já cadastradas por ele | Confirmar a visualização das receitas criadas pelo usuário | Usuário deve estar logado <br> - Navegar até: [Nome Usuário]>Minhas Receitas | - A lista de receitas cadastradas pelo usuário deve ser exibida |
|**RF CT-010.1**| RF-010: O sistema deve permitir que usuário exclua receitas por ele cadastradas | Confirmar a exclusão da receita | O usuário deve estar logado <br> - Navegar até: [Nome Usuário]>Minhas receitas <br> - Selecionar receita a ser excluída e clicar em excluir <br> - Confirmar que deseja excluir <br> - Navegar até: [Nome Usuário]>Minhas receitas | - A receita não deve mais constar na listagem após a exclusão |
|**RF CT-10.2**| RF-010: O sistema deve permitir que usuário exclua receitas por ele cadastradas | Confirmar que apenas o autor da receita é capaz de exclui-la | O usuário pode ou não estar logado <br> - Na página inicial, selecionar uma receita que não seja de autoria do usuário | - A opção de exclusão não deve estar disponível para o usuário não autor |

## Plano de Testes dos Requisitos Não Funcionais
| **Caso de Teste** 	| Requisito Associado 	| Objetivo do Teste 	| Passos	| Critério de Êxito | 
|---|---|---|---|---|
|**RNF CT-001**| RNF-001: O sistema não deve expor dados sensíveis do usuário | Garantir a segurança dos dados sensíveis do usuário | O usuário pode ou não estar logado <br>- Navegar até uma receita<br>- Selecionar o autor<br>- Verificar os dados disponíveis sobre o autor | Apenas o nome de usuário do autor deve estar disponível para visualização |
|**RNF CT-002**| RNF-002: O sistema deve ser desenvolvido utilizando a arquitetura MVC | Verificar se a aplicação foi desenvolvida utilizando a arquitetura MVC | - Revisar código para verificar se todos os módulos correspondem à arquitetura MVC | Devem existir separações claras entre as camadas Model, View e Controller |
|**RNF CT-003**| RNF-003: O sistema deve ser desenvolvido utilizando o framework ASP.NET Core MVC. | Verificar se a aplicação foi desenvolvida inteiramente com framework ASP.NET Core MVC. | - Revisar código para verificar se todos os módulos correspondem à arquitetura MVC e se utilizam do framework ASP.NET Core MVC. <br>| - Ausência de implementação de demais frameworks que não correspondem ao framework ASP.NET Core MVC. |
|**RNF CT-004**| RNF-004: O sistema deve ser capaz de lidar com vários usuários simultâneos sem sofrer degradações significativas no desempenho. | Verificar se a aplicação suporta múltiplos usuários sem diminuição significativa de desempenho. | - Acessar a aplicação por 2 ou mais acessos simultaneamente. <br>        | - A aplicação não apresentou lentidão ou travamentos no carregamento das páginas e nas respostas às ações dos usuários durante o teste. |
|**RNF CT-005**| RNF-005: O sistema deve suportar múltiplos usuários simultaneamente. | Verificar se a aplicação suporta múltiplos usuários. | - Acessar a aplicação por 2 ou mais acessos simultaneamente. <br>        | - A aplicação não caiu durante o teste. |
|**RNF CT-006** 	| RNF-006: O sistema não deve ter tempo médio de resposta superior a 5s. | Verificar se a aplicação ultrapassa o tempo de resposta desejado. | - Acessar a aplicação <br>- Realizar varias ações como: acessar o perfil, visualizar receita, pesquisar uma receita. |  O tempo de Resposta não deve ser superior a 5s em cada pagina. |
|**RNF CT-007**| RFN - 007: O sistema deve ser desenvolvido utilizando práticas recomendadas de clean code. | Garantir que o sistema desenvolvido siga práticas recomendadas de clean code. | - Analisar o código-fonte para identificar possíveis violações às práticas de clean code, como duplicação de código, funções muito longas, nomes de variáveis confusos, etc. |  Ausência de violações graves de clean code. |
|**RNF CT-008**| RNF-008 - O sistema deve ser executado nos navegadores Google Chrome, Opera e Edge.	| verificar se o sistema executa corretamente nos navegadores Google Chrome, Opera e Edge. | - Acessar o sistema utilizando os navegadores Google Chrome, Opera e Edge <br> - Verificar se todas as funcionalidades estão disponíveis e funcionam corretamente em cada navegador <br>        | - O sistema funciona em todos os navegadores testados. |
|**RNF CT-009**| RNF-009 - O sistema deve ser expansivo. | verificar a expansividade do sistema. | - Analisar a estrutura do sistema para verificar se é modular e permite a adição de novas funcionalidades sem alterar o código existente <br> - Testar a adição de novas funcionalidades e verificar se o sistema continua funcionando corretamente. | - O sistema funciona corretamente mesmo após a adição de novas funcionalidades. |
|**RNF CT-010**| RFN - 010: O sistema deve continuar funcionando apresentando conteúdo já carregado em caso de queda de conexão ou instabilidade no banco de dados. | Verificar se a aplicação apresenta o conteúdo já carregado. | - Acessar a aplicação e conseguir visualizar telas anteriormentes acessadas. | -  A aplicação conseguiu exibir as telas acessadas anteriormente durante o teste. |
